# Pulse Channel Configuration

Configuration and MCP tool references for each alert delivery channel.

## Channel Priority (Fallback Order)

When delivering an alert, Pulse attempts channels in this order. If the primary channel fails, it falls through to the next:

1. **Primary channel** (configured by PM during setup)
2. **Secondary channels** (all other enabled channels)
3. **In-system** (always available as last resort)

## Slack

**MCP Tool:** `mcp__3dc7fb47__slack_send_message`

**Configuration:**
```json
{
  "channel": "slack",
  "enabled": true,
  "slack_channel_id": "C0XXXXXXX",
  "slack_channel_name": "#reva-turbo-alerts",
  "message_format": "rich",
  "thread_replies": true
}
```

**Message Format (CRITICAL):**
```
:rotating_light: *CRITICAL: {{ALERT_TYPE}}*

*Entity:* {{ENTITY_ID}} ({{ENTITY_TYPE}})
*Customer:* {{CUSTOMER_NAME}}
*What:* {{ALERT_MESSAGE}}
*Detected:* {{ALERT_TIMESTAMP}}
*Action:* {{ACTION_REQUIRED}}

_Reply with: ack, snooze 1h/4h/1d, or escalate_
```

**Message Format (WARNING):**
```
:warning: *WARNING: {{ALERT_TYPE}}*

*Entity:* {{ENTITY_ID}}
*What:* {{ALERT_MESSAGE}}
*Action:* {{ACTION_REQUIRED}}
```

**Message Format (INFO):**
```
:information_source: {{ALERT_TYPE}}: {{ALERT_MESSAGE}}
```

**Setup steps:**
1. PM provides the Slack channel ID for alerts (or Pulse creates a dedicated #reva-turbo-alerts channel)
2. Verify Slack MCP connection with a test message
3. Configure threading: group related alerts into threads

## iMessage

**MCP Tool:** `mcp__Read_and_Send_iMessages__send_imessage`

**Configuration:**
```json
{
  "channel": "imessage",
  "enabled": true,
  "pm_phone": "+1XXXXXXXXXX",
  "pm_contact_name": "{{PM_NAME}}",
  "critical_only": true
}
```

**Message Format:**
```
REVA-TURBO ALERT [{{PRIORITY}}]
{{ALERT_TYPE}}: {{ALERT_MESSAGE}}
Entity: {{ENTITY_ID}}
Action: {{ACTION_REQUIRED}}
```

**Notes:**
- iMessage is reserved for CRITICAL alerts by default (to avoid notification fatigue on personal devices)
- PM can opt in to WARNING alerts via iMessage during setup
- Messages are kept short for phone notification readability
- No rich formatting (plain text only)

**Setup steps:**
1. PM provides their phone number or Apple ID
2. Verify contact exists via `mcp__Read_and_Send_iMessages__search_contacts`
3. Send a test message to confirm delivery

## Email — Hostinger

**MCP Tool:** `mcp__hostinger-email__send_email`

**Configuration:**
```json
{
  "channel": "hostinger_email",
  "enabled": true,
  "from_address": "alerts@revamfg.com",
  "to_address": "{{PM_EMAIL}}",
  "subject_prefix": "[REVA-TURBO]",
  "include_html": true
}
```

**Subject Line Format:**
```
[REVA-TURBO] [{{PRIORITY}}] {{ALERT_TYPE}} — {{ENTITY_ID}}
```

**Body Format:**
```
REVA-TURBO Alert Notification

Priority: {{PRIORITY}}
Alert: {{ALERT_TYPE}}
Entity: {{ENTITY_ID}} ({{ENTITY_TYPE}})
Customer: {{CUSTOMER_NAME}}
Partner: {{PARTNER_NAME}}

What happened:
{{ALERT_MESSAGE}}

Recommended action:
{{ACTION_REQUIRED}}

Relevant skill: {{RELEVANT_SKILL}}

---
This alert was generated by REVA-TURBO Pulse at {{ALERT_TIMESTAMP}}.
To adjust your alert settings, run /reva-turbo-pulse and select "configure."
```

**Setup steps:**
1. Verify Hostinger email MCP connection
2. Confirm the from address (alerts@revamfg.com or PM's preference)
3. Confirm the to address (PM's email)
4. Send a test email

## Email — Gmail

**MCP Tool:** `mcp__bc6bacab__gmail_create_draft` (for review) or direct send via Gmail API

**Configuration:**
```json
{
  "channel": "gmail",
  "enabled": true,
  "to_address": "{{PM_GMAIL}}",
  "subject_prefix": "[REVA-TURBO]",
  "create_draft_first": false
}
```

**Notes:**
- Same format as Hostinger email
- If `create_draft_first` is true, Pulse creates a Gmail draft instead of sending directly (useful for PMs who want to review before delivery)
- Gmail is typically used as a backup channel if Hostinger is the primary email

**Setup steps:**
1. Verify Gmail MCP connection via `mcp__bc6bacab__gmail_get_profile`
2. Confirm the to address
3. Send a test email

## In-System (Always Active)

**No MCP tool required.** Alerts are stored in `~/.reva-turbo/state/pulse-alerts.jsonl` and presented to the PM on their next REVA-TURBO interaction.

**Configuration:**
```json
{
  "channel": "in_system",
  "enabled": true,
  "show_on_session_start": true,
  "max_pending_display": 10
}
```

**Behavior:**
- On every REVA-TURBO session start, check for pending alerts
- Display up to 10 most recent/highest priority pending alerts
- Provide action options (acknowledge, snooze, escalate, act)
- This channel is always enabled and cannot be disabled (it is the safety net)

## Channel Health Monitoring

Pulse tracks channel delivery success rates:

```json
{
  "channel": "slack",
  "total_attempted": 142,
  "total_delivered": 140,
  "total_failed": 2,
  "last_success": "2026-03-30T14:22:00Z",
  "last_failure": "2026-03-28T03:15:00Z",
  "failure_reason": "MCP connection timeout"
}
```

If a channel's failure rate exceeds 10% over 7 days, alert the PM:

> "Slack delivery has failed 15% of the time this week. Check your Slack MCP connection. Alerts are being routed to your backup channel (email) in the meantime."
